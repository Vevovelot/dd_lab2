# Правила работы над проектом

Эти правила обязательны и имеют приоритет над любыми твоими предпочтениями.

## Процесс
- Работай строго по вехам в порядке их следования. НЕ начинай веху N+1, пока веха N не завершена и не подтверждена мной.
- После завершения каждой вехи ОСТАНОВИСЬ. Покажи, что сделано, как это проверить, и жди моего подтверждения.
- Перед началом вехи кратко опиши план (3-5 пунктов) и дождись «ок».
- Не реализуй функциональность будущих вех «про запас».

## Код
- Целевой фреймворк: net10.0. Не меняй версию без моего разрешения.
- НЕ добавляй NuGet-пакеты без явного согласования. Сначала спроси, зачем пакет и какие альтернативы.
- Разрешённые пакеты: Microsoft.Data.Sqlite, Microsoft.Extensions.Configuration, xUnit, NSubstitute. Всё остальное — через запрос.
- Используй HttpClient и System.Text.Json.
- Код должен оставаться кроссплатформенным. НЕ используй платформо-зависимые пути и API. Пути собирай только через Path.Combine.
- Используй единый проект для всего решения.
- Никаких хардкодов base_url, имена модели — только из конфигурации appsettings.json, api_key - из переменной окружения.
- Работа с моделью идёт только через интерфейс провайдера (IModelProvider). 
  Не обращайся к конкретному провайдеру напрямую из логики агента.
- Выбор провайдера и его параметры — только в appsettings.json. 
  Никаких хардкодов под конкретного провайдера.

## Безопасность (действует с первой вехи, не только в Вехе 6)
- Файловые инструменты НЕ должны писать или удалять за пределами рабочего каталога проекта. Проверяй это в коде.
- Не запускай произвольные shell-команды для «проверки» без моего ведома.

## Проверка
- После изменений код должен собираться: `dotnet build`.
- Если добавил логику — добавь тест и покажи, что `dotnet test` проходит.
- Если что-то не работает или ты не уверен — СКАЖИ об этом прямо, не выдавай предположение за факт.

## Стиль ответов
- Будь краток. Не пересказывай то, что и так видно в коде.
- Если правило выше мешает выполнить мою просьбу — скажи об этом, а не нарушай молча.

# Описание проекта
Это учебный проект по разработке кодинг-агента. Рабочее название - iCode.
Цель - научиться разрабатывать агентов для нейронных сетей.
Результатом должен быть агент, который принимает задачу на естественном языке, сам читает и правит файлы, запускает команды, работает в цикле до результата.

## Определения
Проект - любой каталог файловой системы из которого запущен агент.
Рабочий каталог проекта, рабочий каталог - каталог запуска агента.
project_name - уникальное имя каталога, создаваемого агентом в своём служебном каталоге, для хранения данных проекта, с которым работает агент. project_name формируется путём хэширования полного пути рабочего каталога.
HOME_FOLDER - домашний каталог пользователя.

# Порядок разработки.
Разработка разбита на вехи, перечисленные ниже. Реализацию каждой вехи проверять с помощью тестов.

### Веха 1. Цикл агента
Цикл - получить задачу → спросить модель → модель просит вызвать инструмент → выполнить → результат обратно в контекст → повторить, пока задача не решена.
На данном этапе без инструментов: простой консольный диалог в цикле, где на каждом ходу вся история собирается заново и уходит модели.
Контекст собирать в <HOME_FOLDER>/.iCode/<project_name>/context - файл sqlite3 базы данных.
Схема таблицы контекста: timestamp, role, content. Схема может уточняться по ходу реализации проекта.
Реализовать окончание цикла / завершение работы агента по компанде '/exit'.
Дополнительно к циклу:
- После каждого ответа модели читать usage из ответа (input/output токены) и накапливать размер контекста.
- ContextWindow для активного провайдера задаётся в appsettings.json.
- При приближении к лимиту (например, >80% ContextWindow) выводить предупреждение пользователю.
- Ошибку переполнения контекста от провайдера перехватывать и показывать внятным сообщением, не падать.
- Команда /context — показать текущий размер контекста (токены) и процент от ContextWindow.
Тестирование данной вехи - только накопление контекста и завершения диалога.
Конец вехи - программа, позволяющая вести диалог с пользователем с накоплением истории сообщений (контекста).

### Веха 2. Инструменты
Реализовать поочерёдно следующие инструменты: Read (чтение файла), Write (создание нового файла), Update (редактирование существующего файла), Delete, List (перечисление файлов в рабочем каталоге), Execute (команда в терминале). Модель, с которой работает агент, сама решает, какой инструмент вызвать и с какими аргументами. Код агента выполняет и возвращает результат обратно в цикл.
В случае ошибки возвращать её в модель.
Реализацию сделать OpenAI совместимой (tool use / function calling).
Все инструменты должны выполняться только в рабочем каталоге и подкаталогах. Выход на уровни выше рабочего каталога запрещены и должны ограничиваться со стороны агента. Предусмотреть защиту от использования '..' и символических ссылок.
Конец вехи - реализация всех перечисленных инструментов.

### Веха 3. AGENTS.md — контекст проекта
Агент, сразу после запуска должен читать файл AGENTS.md из рабочего каталога и класть его в системный промпт.
Конец вехи - указания AGENTS.md сохраняются в системном промпте.

### Веха 4. Skills — знание по требованию
Агент должен читать каталог SKILLS в рабочем каталоге проекта, содержащий файлы со скиллами - короткое описание (когда его применять) и тело (как делать что-то конкретное). Агент видит только описания; тело нужного скилла он подгружает в контекст по требованию, а не держит всё сразу.
Формат файла скилла - Markdown с фронтматтером.
Конец вехи - описания каждого скилла сохраняются в системном промпте, подгрузка происходит по требованию модели.

### Веха 5. Субагент — изолированный контекст
Реализовать инструмент, который запускает субагента — отдельный экземпляр агента с чистым контекстом — на подзадачу и возвращает наружу только результат. Например: «изучи структуру проекта и верни краткий отчёт».
Субагент работает с теми же инструментами, что и агент.
Субагент при старте ведёт себя также как агент.
На субагента накладываются те же ограничения, что и на агента.
Субагент не может запускать других субагентов.
Конец вехи - работающая функциональность запуска субагента и передачи результата.

### Веха 6. Разрешения и полировка
Реализовать подтверждение перед опасными действиями: Execute, Delete, Write. Варианты подтверждения для действий - разрешить однократно, разрешить до конца сессии, разрешить всегда. Перечень выданных разрешений сохранять в <HOME_FOLDER>/.iCode/<project_name>/permissions
Конец вехи - работающая функциональность подтверждения опасных действий и запрет их выполнения при отсутствии подтверждения.
